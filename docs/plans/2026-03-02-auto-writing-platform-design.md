# 自动写作平台设计方案

## 1. 项目目标

本项目要从零做成一个可直接售卖、可长期运营的正式产品，不是演示版，也不是半成品。

用户在同一个页面内完成整条流程：
- 上传多个文件
- 填写特殊要求
- 等待系统分析
- 查看并修改大纲
- 确认后生成正文
- 下载最终版文章和核验报告
- 可选继续使用自动降 AI
- 查看人工降 AI 客服入口

产品同时满足以下业务目标：
- 支持登录、充值、订阅、扣费
- 支持国内和海外收款
- 支持后台统一管理用户、订单、任务、额度和文件
- 用户内容只保留 3 天，账务和运营记录长期保留
- 后续可持续扩展，不因前期架构偷懒导致重做

## 2. 核心原则

### 2.1 必须保留的业务规则

- 不能只靠文件名猜内容，必须读取每个上传文件的实际内容
- 任务要求文件中的硬要求优先级最高
- 如果任务要求中明确写了字数或引用格式，必须按它执行
- 只有在任务要求里没有明确写时，才使用默认值
- 默认字数为 `2000`
- 默认引用格式为 `APA 7`
- 用户填写的特殊要求必须在第一次生成大纲时就生效
- 如果多个文件都像任务书，系统必须判断清楚；如果无法稳定判断，必须在当前页面让用户确认
- 大纲必须先给用户确认，不能直接跳过大纲去写正文
- 大纲确认可反复修改，但必须设置次数上限，防止无限循环
- 正文必须是英文，必须包含标题、章节结构和 `References`
- 正文字数校正只处理正文部分，不把 `References` 计算进去
- 核验报告不假装读取文献全文，只根据标题、摘要、年份、DOI 或链接等可获得信息判断是否基本对应
- 自动降 AI 只处理正文，不处理参考文献

### 2.2 产品体验原则

- 用户从开始到结束都停留在同一个工作台页面
- 页面不跳来跳去，但后台可以分步骤慢慢执行
- 每一步都显示当前进度和状态，让用户知道系统做到哪里
- 每一步的中间结果都保留在系统记录中，便于恢复、追踪和重试

## 3. 推荐技术方案

本项目采用“托管式一体化正式架构”。

### 3.1 前台与接口

- `Next.js`

用途：
- 做用户看到的网站页面
- 做后台接口
- 保持前台和后端在同一个项目里，减少维护复杂度

原因：
- 适合做登录、上传、支付、任务状态展示、后台管理
- 社区成熟，适合长期维护

### 3.2 部署平台

- `Vercel`

用途：
- 托管网站
- 统一管理部署、域名、环境变量和线上版本

原因：
- 比自己租服务器省心
- 更适合不会写代码但希望后续维护简单的运营场景

### 3.3 登录、数据库、文件存储

- `Supabase`

用途：
- 用户注册和登录
- 存用户资料、余额、订单、任务、日志
- 存上传文件和输出文件

原因：
- 登录、数据库、文件存储整合在一套平台里
- 后期管理简单

### 3.4 长流程后台任务

- `Trigger.dev`

用途：
- 跑长时间任务
- 把“读取文件 -> 分析要求 -> 生成大纲 -> 等待确认 -> 写正文 -> 校字数 -> 核验引用 -> 导出文件”拆成可追踪的后台步骤

原因：
- 避免一次请求跑太久导致超时
- 方便失败重试
- 方便用户在页面持续看到状态

### 3.5 AI 核心能力

写作与分析：
- `OpenAI API`

用途：
- 读取并理解上传文件内容
- 判断任务要求文件与背景材料
- 提取写作规则
- 生成英文大纲和中文对照
- 按确认后的大纲写英文正文
- 做字数校正
- 做引用核验说明

说明：
- 本方案不引入 `Unstructured API`
- 文件处理仍由系统单独做“文本提取步骤”，只是 AI 核心统一使用 `OpenAI API`
- 不把所有原文件在每一步都重复整份输入模型，避免成本和速度失控

自动降 AI：
- `StealthGPT API`

用途：
- 只用于“自动降 AI”阶段
- 只处理正文，不处理 `References`
- 输出用于生成“降AI后版本（Word）”

原因：
- 有现成 API，可直接对接
- 按字数计费，成本更容易预估
- 专门做人性化改写，结果比直接用通用写作模型改写更稳定

说明：
- 自动降 AI 不再使用 `OpenAI API` 自行改写
- 为提高稳定性，长正文应按章节或分段拆开后再调用 `StealthGPT API`

### 3.6 文档输出

- `python-docx`
- `reportlab`

用途：
- `python-docx` 生成最终版 Word
- `reportlab` 生成引用核验 PDF

原因：
- 输出稳定，适合固定排版
- 可精确控制字体、间距、缩进和结构

### 3.7 支付系统

海外支付：
- `Stripe`
- 手动稳定币收款（`USDT` / `USDC`）

国内支付：
- `支付宝`
- `微信支付`

说明：
- 系统内部统一收口为“支付成功后给用户账户增加额度”
- 前台可以有多种支付方式，后台只维护一套额度账本
- 稳定币收款改为“用户按地址转账 -> 运营人工确认到账 -> 手动补额度”
- 稳定币支持 `Base / Ethereum / Solana`
- 国内支付在代码设计上支持官方直连，但落地时优先考虑合规统一服务商，以降低维护成本和对账复杂度

## 4. 产品结构

### 4.1 用户端结构

用户端建议采用单工作台体验，并配合少量辅助页面。

主要页面：

1. 开始写作
- 上传多个文件
- 输入特殊要求
- 显示预计消耗额度
- 发起任务

2. 我的任务
- 查看近 3 天任务
- 查看当前进度
- 下载生成结果
- 查看是否已过期

3. 充值中心
- 购买充值包
- 开通订阅
- 查看余额

4. 账户中心
- 查看账户信息
- 查看消费记录
- 查看当前套餐

### 4.2 后台管理结构

后台是运营后台，不是面向程序员的技术控制台。

后台首页建议提供 6 个入口：

1. 用户管理
- 搜索用户
- 查看余额、冻结额度、累计充值、累计消耗
- 冻结或解冻账户
- 手动补额度或扣额度

2. 订单与收款
- 查看每笔订单
- 查看支付方式和支付状态
- 手动补单
- 处理退款后的额度同步

3. 任务管理
- 查看任务当前步骤
- 查看失败原因
- 手动重试任务
- 查看本次输入和输出摘要

4. 文件与交付
- 查看生成文件
- 查看下载次数
- 查看过期时间
- 手动延长保留时间
- 手动提前删除

5. 套餐与价格
- 设置充值包
- 设置订阅包
- 设置价格
- 设置赠送额度
- 开关不同支付方式

6. 成本与利润
- 统计模型成本
- 统计降 AI 成本
- 粗略查看收入和利润

## 5. 端到端业务流程

### 5.1 提交任务

用户在同一个页面内完成：
- 上传多个文件
- 填写特殊要求
- 查看本次预计消耗额度
- 点击开始分析

系统执行：
- 检查余额是否足够
- 如果足够，先冻结本次所需额度
- 创建任务记录
- 上传原始文件到存储
- 进入后台任务队列

### 5.2 文件读取与文本提取

系统必须逐个读取文件内容，而不是只看文件名。

支持格式：
- `txt`
- `md`
- `docx`
- `pdf`
- `ppt`
- `pptx`

处理原则：
- 每个文件都生成一份“提取文本”
- 所有后续分析尽量基于提取后的文本进行
- 文件提取失败时记录失败原因，并允许用户看到哪个文件有问题

### 5.3 文件分类

系统把文件分成三类：
- 任务要求文件
- 背景材料
- 基本无关

分类目标：
- 找出硬要求
- 找出评分重点
- 找出推荐框架
- 找出要回答的问题
- 找出可引用的背景信息

多任务书冲突处理：
- 如果模型无法稳定判断唯一主任务文件，则在当前页面提示用户确认
- 用户确认前，流程停在当前状态，不进入大纲阶段

### 5.4 生成写作规则卡

系统在内部生成一张“写作规则卡”，作为后续全部内容生成的统一依据。

规则卡包含：
- 主题
- 目标字数
- 引用格式
- 明确章节要求（如果存在）
- 必须回答的问题
- 评分重点
- 用户特殊要求
- 可优先使用的背景材料

优先级固定为：
1. 任务要求文件中的硬要求
2. 用户输入的特殊要求
3. 背景材料中的有效信息
4. 系统默认值

默认值仅在没有明确要求时使用：
- 字数：`2000`
- 引用格式：`APA 7`

### 5.5 生成大纲

系统先生成英文大纲，再尽量生成中文对照。

英文大纲必须包含：
- 独立文章标题
- 与目标字数匹配的章节数
- 每章主要内容说明
- 每章 3 到 5 个重点

规则：
- 默认每 `500` 字增加一个章节
- 如果任务要求或用户明确指定章节数，则按指定值
- 短篇一般给 3 个重点
- 中等篇幅一般给 4 个重点
- 更长篇幅最多给 5 个重点
- 用户说“短一点”“简一点”时，下一版应减少章节复杂度和每章重点数量

中文对照规则：
- 只用于给中文用户阅读理解
- 不参与后续正式写作
- 如果中文对照生成失败，不阻塞英文大纲继续使用

页面展示必须同时明确告知：
- 本次目标字数
- 本次引用格式

### 5.6 大纲确认与修改

用户可以：
- 查看英文主版大纲
- 查看中文对照
- 输入修改意见
- 确认继续写作

系统规则：
- 每次重生大纲都保留文章标题
- 每次重生大纲都更新章节结构
- 用户说“短一点”时，大纲必须真的变短，而不是复述用户原话
- 为防止无限循环，默认最多允许 `5` 次免费大纲重生

达到上限后的处理：
- 提示用户当前已到本任务大纲修改上限
- 用户可选择继续写正文
- 或者发起新任务

### 5.7 生成正文

只有在用户确认大纲后，系统才开始生成正文。

正文规则：
- 必须为英文
- 必须是完整文章，不能是要点列表
- 必须保留独立文章标题
- 必须有清晰的章节结构
- 必须包含 `References`
- 必须使用第三人称
- 必须围绕中心论点展开
- 必须包含证据支撑和批判性分析
- 必须遵守已锁定的引用格式

系统允许在此步骤接入额外系统提示词，以强化写作规则，但不能覆盖前面已经锁定的“写作规则卡”。

### 5.8 字数校正

初稿生成后，不直接交付，先执行字数校正。

字数校正规则：
- 只统计正文，不统计 `References`
- 按目标区间缩短或补足正文
- 允许多轮尝试
- 尽量把正文拉回目标范围

为避免用户误解，系统采用“正式版 + 候选版”机制：
- 当前正式版：当前已通过检查的版本
- 新候选版：本轮刚调整出的版本

如果候选版字数更接近目标，但结构检查未通过：
- 前台提示“新版本已生成，但暂未替换正式版”
- 避免用户误以为字数没有变化

### 5.9 引用核验

系统对参考文献逐条生成核验说明，但不假装读取全文。

核验依据：
- 文献标题
- 摘要
- 年份
- 作者信息（如可得）
- DOI
- 链接

核验结论分为两类：
- 基本可对应
- 有风险

判定规则：
- 若标题、摘要与正文引用主张基本一致，且 DOI 或链接信息无明显冲突，则为“基本可对应”
- 若标题、摘要、年份、链接明显矛盾，或关键字段缺失严重，则为“有风险”

每条核验都必须附带文字说明，不能只给标签。

### 5.10 交付包装

系统在导出前先做结构补齐。

最低结构要求：
- 独立标题
- 章节标题
- `References`

若缺结构部件：
- 优先补格式与结构
- 不直接整篇重写
- 尽量不改变原正文含义

## 6. 输出文件规范

### 6.1 最终版文章（Word）

输出工具：
- `python-docx`

固定排版要求：
- 字体：`Times New Roman`
- 字号：`12`
- 颜色：黑色
- 文章标题：居中
- 段落之间：保留空行
- 小标题与正文之间：保留空行
- `References`：全部使用悬挂缩进

说明：
- 导出层只负责排版，不负责改写正文
- 若结构齐全，仅修正排版，不重写内容

### 6.2 核验报告（PDF）

输出工具：
- `reportlab`

固定内容建议：
- 封面页
- 报告编号
- 生成时间
- 本次任务基本信息
- 逐条文献核验结果
- 总结说明

质量要求：
- 无重叠
- 无乱码
- 无错位
- 版面清晰，便于直接下载交付

### 6.3 自动降 AI 后版本（Word）

用户点击“自动降AI”后：
- 只处理正文内容
- 不处理参考文献
- 完成后生成新的 Word 文件

结果区新增内容：
- `降AI后版本（Word）` 下载按钮
- 一个固定显示的微信二维码卡片

二维码卡片固定文案：
- `人工降ai 请联系客服`

用途：
- 作为自动服务之外的人工服务引流入口

## 7. 存储与数据保留

### 7.1 保留 3 天的内容

以下内容仅保留 3 天：
- 用户上传的原始文件
- 提取后的文本
- 大纲各版本
- 正文草稿
- 最终版 Word
- 核验报告 PDF
- 降 AI 后 Word

### 7.2 长期保留的记录

以下内容长期保留，不随 3 天清理一起删除：
- 用户账号
- 订阅记录
- 余额记录
- 冻结额度记录
- 消费记录
- 订单记录
- 支付状态
- 任务状态
- 成本统计
- 操作日志

### 7.3 删除机制

每个任务完成后记录一个到期时间：
- 到期时间 = 完成时间 + 3 天

定时清理任务执行：
- 到点后删除文件和可下载内容
- 保留任务摘要记录
- 前台下载按钮变为“已过期”
- 后台仍可看到该任务存在过，但文件已清理

## 8. 额度、扣费与记账

### 8.1 额度模型

用户账户使用“额度”作为统一消费单位。

但系统内部必须拆成两个额度桶：
- 充值额度
- 订阅月度额度

这样做的原因：
- 充值额度是用户自己买的，不应被月底清零
- 订阅月度额度是当期权益，可以按周期重发和清零

月底清零规则：
- 只清空未用完的“订阅月度额度”
- 不清空“充值额度”

### 8.2 扣费项目

系统至少拆成两个收费动作：

1. 文章生成
- 从文件分析到最终版文章与核验报告，算一次完整生成
- 按目标字数区间扣费

2. 自动降 AI
- 单独扣费
- 按字数区间扣费

### 8.3 扣费时机

为避免争议，采用“先冻结，后正式扣除”的方式。

执行规则：
- 用户点击开始分析时，先检查余额
- 余额足够则冻结本次预计额度
- 任务成功后，正式扣除冻结额度
- 任务失败则释放冻结额度

自动降 AI 同理：
- 开始前冻结
- 成功后扣除
- 失败则退回

## 9. 支付与售卖模式

### 9.1 支付入口

支持以下支付方式：
- `Stripe`
- `Coinbase Commerce`
- `支付宝`
- `微信支付`

内部统一逻辑：
- 支付成功
- 订单记录更新
- 给对应账户增加额度

### 9.2 售卖模式

推荐双轨制：

1. 充值包
- 用户一次性购买
- 获得一批额度
- 用完再充

2. 月订阅
- 用户按月付费
- 每月自动发放额度
- 月度额度不滚存
- 到月底或当前周期结束时，未用完的订阅月度额度自动清零
- 到期自动续费

建议前期只做 3 档，避免套餐过多：
- 基础
- 进阶
- 高级

### 9.3 订阅生命周期

系统需要单独保存每个订阅的生命周期信息，包括：
- 用户
- 订阅平台中的订阅 ID
- 对应套餐
- 当前状态
- 当前周期结束时间

同步规则：
- 接收 `Stripe` 的订阅创建、更新、取消事件
- 把订阅状态同步到本地数据库

月度发放规则：
- 使用 `Trigger.dev` 每天检查订阅
- 只有状态仍然有效的订阅才发放新的月度额度
- 每个自然月或每个新的订阅周期只发放一次，防止重复发放

## 10. 任务状态设计

系统采用状态机来管理整条流程。

推荐状态：
- 已创建
- 已冻结额度
- 正在提取文件
- 等待确认主任务文件
- 正在生成写作规则
- 已生成大纲
- 等待用户确认大纲
- 正在写正文
- 正在校正字数
- 正在核验引用
- 正在导出文件
- 可下载
- 自动降 AI 处理中
- 自动降 AI 已完成
- 失败
- 已过期

作用：
- 前台清楚显示进度
- 后台清楚定位卡点
- 失败时可按步骤重试

## 11. 错误处理原则

系统不能把所有问题都当成整单失败。

分级处理建议：

可降级继续的情况：
- 中文对照生成失败
- 某条参考文献核验存在风险
- 个别背景材料提取失败，但主任务文件正常

必须暂停等待用户的情况：
- 多个文件都像主任务文件，无法稳定判断
- 用户余额不足
- 用户必须确认大纲但尚未确认

必须标记失败的情况：
- 主任务文件无法读取
- 正文生成失败且重试后仍失败
- 导出文档失败且重试后仍失败
- 支付回调异常导致账务无法确认

## 12. 安全与运营要求

- 所有文件访问都必须绑定所属用户，防止用户互相看到别人的文件
- 文件下载链接应使用带时效的安全链接
- 后台必须保留关键操作日志，便于追踪补额度、退款、重试等操作
- 后台字段尽量使用中文，方便运营使用
- 关键业务动作应有“确认提示”，避免误操作

## 15. 设计结论

本项目应采用“托管式一体化正式架构”，核心目标不是最快做出一个演示品，而是直接做成可售卖、可扩展、可运营的正式系统。

最终定稿架构如下：
- 前台与接口：`Next.js`
- 部署：`Vercel`
- 登录 / 数据库 / 存储：`Supabase`
- 长流程任务：`Trigger.dev`
- 写作 AI：`OpenAI API`
- 自动降 AI：`StealthGPT API`
- Word 导出：`python-docx`
- PDF 报告：`reportlab`
- 海外支付：`Stripe` + `Coinbase Commerce`
- 国内支付：`支付宝` + `微信支付`

业务核心设计如下：
- 用户始终在同一个页面完成任务
- 大纲先确认，再写正文
- 内容文件仅保留 3 天
- 账务与运营记录长期保留
- 订阅额度与充值额度分开记账
- 自动降 AI 与文章生成分开扣费
- 自动降 AI 完成后同时展示下载入口与人工降 AI 微信二维码入口

这套设计满足当前已确认的全部业务规则，并为后续实施计划提供固定边界。
