# 拼代代PDD 前端合并与激活码模式替换设计方案

## 1. 目标

本次改造要完成两件事：

1. 把新的“拼代代PDD”品牌前端视觉，合并进当前正式主项目。
2. 把用户端的业务口径，从“在线支付 / 套餐 / 订阅”统一切换成“额度激活码 + 固定积分扣费”。

本次设计的重点，是先把用户能看到的页面和规则统一正确，再在后续实施里逐步替换底层逻辑。

## 2. 当前现状

当前有两套前端来源：

1. 正式主项目（Next.js）
- 路径：`/Users/jeffo/Desktop/自动写作/.worktrees/bootstrap-shell`
- 已包含：登录保护、工作台骨架、任务页、后台管理、下载保护、任务流程与接口骨架
- 优势：后端能力已经在这里，适合作为正式主站继续扩展

2. 新视觉前端（Vite + React）
- 路径：`/Users/jeffo/Desktop/自动写作/拼代代-pdd`
- 已包含：销售型首页、品牌化登录页、注册页、额度页、任务页、工作台的高质量视觉壳子
- 优势：页面审美和销售感更强，适合作为视觉模板来源

同时，用户已经确认新的业务规则：
- 不再做在线收款
- 不再做按月时长激活
- 用户登录后账号长期有效
- 通过额度激活码充值积分
- 激活码只能使用一次
- 同一账号可多次兑换不同激活码
- 四档固定积分：`1000 / 5000 / 10000 / 20000`
- 生成文章固定扣 `500`
- 自动降AI固定扣 `500`

## 3. 推荐方案

推荐采用“主项目保留，视觉迁移合并”的方案。

具体来说：
- 保留 Next.js 主项目作为唯一正式主站
- 不把 Vite 项目作为正式运行时前端
- 以 Vite 项目为视觉和交互模板来源
- 把 Vite 项目的页面结构、品牌组件、展示模块，迁移到 Next.js 主项目对应路由

这样做的原因：
- 不破坏现有的任务流程、接口和登录保护
- 不需要维护两套前端运行栈
- 视觉和业务能逐步合并，风险可控
- 后续做激活码系统时，可以直接接到主项目现有后端

## 4. 品牌统一规则

### 4.1 品牌名称

全站统一品牌名称为：
- 中文展示：`拼代代PDD`
- 英文标识：`Pin Dai Dai PDD`

所有旧名称、旧占位名、生成器默认名都不再使用。

### 4.2 品牌素材

用户已提供真实素材：
- Logo：`logo.jpg`
- 客服二维码：`qrcode.jpg`

在正式主项目中，应将这两份素材复制到主项目自己的 `public` 目录内，避免依赖外部文件路径。

### 4.3 品牌露出位置

品牌必须在全站核心页面稳定出现，但不能廉价地“乱贴”。

推荐固定位置：
- 顶部导航左上角：主品牌位
- 页脚：次品牌位
- 登录页 / 注册页：大品牌头图
- 额度中心：品牌头图 + 销售区品牌位
- 工作台 / 我的任务：页面头部品牌识别
- 首页：首屏品牌位 + 销售转化区品牌位

## 5. 用户端页面改造方案

### 5.1 首页 `/`

首页改造成正式销售页，直接替换当前简单入口页。

必须包含：
- Hero 首屏
- 信任条
- 适用人群
- 核心流程
- 功能矩阵
- 匿名成功案例
- 用户反馈
- 服务标准
- 激活码规则说明
- 联系销售区（放真实二维码）
- FAQ

首页的职责不再是“跳转入口”，而是：
- 建立品牌
- 建立信任
- 解释产品
- 促成用户联系销售购买激活码

### 5.2 登录页 `/login`

保留现有路由，但将页面换成品牌化版本。

目标：
- 使用真实 logo
- 文案统一成拼代代PDD
- 与首页保持统一视觉
- 保留联系客服入口

### 5.3 注册页 `/register`

主项目新增正式注册页。

目标：
- 提供注册入口
- 提示“注册后账号长期有效”
- 提示“通过额度激活码充值积分”

### 5.4 额度中心 `/billing`

保留技术路由 `/billing`，但完全改造其内容和含义。

改造后用户看到的，不再是支付页面，而是：
- 当前积分
- 激活码输入
- 兑换规则
- 四档额度说明
- 最近兑换记录
- 最近消耗记录
- 联系销售区
- 真实客服二维码

所有在线支付入口全部从该页面下线。

### 5.5 友好别名路由 `/recharge`

新增用户更易理解的页面入口 `/recharge`。

实现方式可为：
- 直接渲染与 `/billing` 相同内容
或
- 跳转到 `/billing`

推荐直接复用同一组件，以减少重复。

### 5.6 工作台 `/workspace`

保留现有业务骨架和任务流程结构，但整体视觉迁移为新的品牌版风格。

必须保留：
- 多文件上传
- 特殊要求
- 本次预计消耗
- 任务进度
- 大纲确认
- 交付结果
- 自动降AI
- 人工降AI客服入口

同时把“固定 500 积分”的新规则明确展示出来。

### 5.7 我的任务 `/tasks`

保留现有路由，换成更正式的列表型展示。

页面应强化：
- 任务标题
- 状态
- 创建时间
- 消耗积分
- 下载入口
- 失败任务重试提示

## 6. 旧支付相关功能的处理原则

### 6.1 立即下线的用户端内容

以下内容立即从用户端页面中移除：
- Stripe
- 支付宝付款
- 微信付款
- 加密货币付款
- 月订阅入口
- 美元价格套餐
- “支付成功 / 下单成功”类文案

这些不再出现在任何用户前端页面中。

### 6.2 立即保留但暂不删除的底层能力

以下底层代码暂时不物理删除：
- 旧支付接口
- 旧支付订单结构
- 旧订阅结构
- 旧支付回调逻辑

原因：
- 先做页面合并更稳
- 先冻结使用，后面再逐步替换
- 避免在视觉合并阶段引发大面积回归问题

## 7. 积分和激活码模型

### 7.1 用户视角规则

用户看到的规则统一为：
- 账号长期有效
- 靠激活码充值积分
- 每个激活码只能兑换一次
- 同账号可反复兑换多个不同激活码
- 生成文章：`500`
- 自动降AI：`500`

### 7.2 后续底层替换方向

后续正式实施时，旧的“充值额度 + 订阅额度”双轨结构将逐步让位于更直观的积分模型：
- 当前总积分
- 已冻结积分（如保留）
- 最近兑换记录
- 最近消耗记录

同样，旧的“按字数分档扣费”将替换为固定扣费。

## 8. 后台改造方向

后台后续应从“支付后台”切换成“激活码运营后台”。

重点能力应变为：
- 生成激活码
- 批量生成激活码
- 查看激活码是否已使用
- 查看激活码对应档位
- 查看哪个用户兑换了哪个码
- 手动补积分 / 扣积分
- 查看积分消耗记录

## 9. 实施顺序建议

推荐按以下顺序实施：

1. 先迁移品牌和页面外观
2. 先统一用户端文案和业务口径
3. 再替换固定积分扣费逻辑
4. 再新增激活码系统
5. 再调整后台中控为“发码和管码”
6. 最后再处理旧支付底层的冻结或移除

这样可以避免“一边改前端、一边拆底层”导致的连锁问题。

## 10. 风险与控制

### 10.1 最大风险

最大的风险不是页面，而是：
- 前端已经是激活码模式
- 底层仍然还是支付和订阅逻辑

如果不按阶段推进，就会出现“页面说一套，后台逻辑还是另一套”的混乱。

### 10.2 控制方式

本次设计要求后续实施遵循：
- 先改页面，再改业务
- 先隐藏旧支付入口，再替换底层
- 每一阶段都先跑检查，再继续下一阶段
- 不在同一轮里同时大改页面、数据库和计费逻辑

## 11. 成功标准

本轮完整改造完成后，成功标准应为：

1. 用户看到的是完整统一的拼代代PDD品牌站
2. 首页已经是可售卖、可引导成交的正式销售页
3. 用户端所有“在线支付”痕迹消失
4. 额度中心清晰展示激活码规则和积分规则
5. 工作台和任务页视觉完成升级，但原有主流程不被破坏
6. 后续可以在不推翻页面的情况下，继续接入真实激活码系统
