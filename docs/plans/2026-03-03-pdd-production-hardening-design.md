# 拼代代PDD 正式版补完设计方案

## 1. 背景

当前线上已经部署的是拼代代PDD的 Next.js 主项目，前端视觉样式、品牌元素和页面结构已经符合预期，必须保持不变。

但代码审查结果表明，当前版本仍然存在明显的“演示逻辑”残留：

- 登录与注册页面仍是前端假跳转，没有真正创建或建立登录会话
- 工作台与额度页展示的是写死积分，不是数据库真实余额
- 激活码兑换与下载接口仍依赖前端传 `userId`，存在安全问题
- 任务创建、主任务文件确认、大纲确认等关键接口仍为 501 占位
- 管理后台权限仍靠普通 cookie 判断，后台数据大量来自 mock
- 文件提取与文章生成链路仍只有骨架，尚未形成真实可交付闭环

本设计的目标不是推翻重写，而是在**不改变当前前端样式与交互外观**的前提下，把现有项目补成可上线运营的正式产品。

## 2. 不可变约束

### 2.1 必须保留

- 当前部署在 Vercel 的 Next.js 主项目作为唯一主项目
- 当前前端视觉设计、页面结构、品牌文案风格
- 品牌名称“拼代代PDD”
- 当前 Logo 与客服二维码素材
- 用户端核心路由：`/login`、`/register`、`/workspace`、`/billing`、`/tasks`、`/admin`

### 2.2 必须遵守的业务规则

- 用户长期登录，不做时长型激活码
- 充值方式只保留“额度激活码”
- 激活码固定四档：`1000 / 5000 / 10000 / 20000`
- 每次生成文章扣 `500` 积分
- 每次自动降AI扣 `500` 积分
- 用户文件与交付文件默认保留 `3` 天
- 核验报告不区分“部分支持 / 完全确认”，只根据标题、年份、DOI、链接、摘要等公开信息判断是否基本对应
- 自动降AI必须直接走 StealthGPT，只处理正文，不处理 `References`
- 用户必须在同一个工作台页面里完成主要流程

## 3. 目标状态

项目补完后，系统应具备以下正式版能力：

### 3.1 账号与额度

- 用户可通过邮箱密码真实注册、登录、退出
- 注册后自动创建用户资料与积分钱包
- 登录后工作台与额度中心实时显示当前余额
- 激活码兑换只基于当前登录用户，不再接受前端传入 `userId`
- 激活码兑换结果、积分流水、消耗流水可在后台查询

### 3.2 写作主流程

- 用户上传多个文件并填写特殊要求
- 系统读取文件正文，不只看文件名
- 系统识别主任务文件、背景材料、无关材料
- 系统生成规则卡，再生成英文大纲与中文对照
- 用户可确认或修改大纲，最多允许有限次修改
- 用户确认后再生成正文、执行字数校正、引用核验、交付导出
- 最终交付 Word 正文、PDF 核验报告、可选降AI后的 Word

### 3.3 安全与管理

- 后台权限以数据库角色为准，不再依赖伪造 cookie
- 所有下载接口按当前登录用户校验归属
- 积分扣减、冻结、解冻在服务端原子执行
- 文件 3 天自动过期，但任务摘要、账务、日志长期保留
- 管理员可查看用户、任务、文件、激活码、流水并进行有限人工处理

## 4. 总体架构

### 4.1 单体应用架构

继续采用单个 Next.js 项目承载：

- 用户页面
- 管理后台
- 业务接口
- 文件导出触发
- 鉴权与会话管理

原因：

- 对当前阶段最稳，维护成本最低
- 不引入第二套独立后端，避免未来排查困难
- 能在不改前端设计的前提下直接补全后端逻辑

### 4.2 基础设施

- 页面与接口部署：Vercel
- 认证、数据库、对象存储：Supabase
- 写作与分析：OpenAI API
- 自动降AI：StealthGPT API
- Word 导出：`python-docx` worker
- PDF 导出：`reportlab` worker

### 4.3 异步执行策略

当前实现先补齐“可运行闭环”，再逐步强化为明确的后台任务链。

第一阶段：

- 在现有项目内完成可靠的状态推进、数据落库、失败处理与恢复

第二阶段：

- 把耗时步骤稳定迁移到 Trigger.dev 编排
- 页面只轮询状态，不承担长耗时执行

这样可以避免在最关键的产品补完阶段，同时重构太多基础设施。

## 5. 认证与会话设计

### 5.1 登录与注册

- `/register` 使用 Supabase Auth 创建账号
- `/login` 使用 Supabase Auth 建立真实登录会话
- 注册成功后自动登录并进入 `/workspace`
- 登录成功后进入跳转目标或默认进入 `/workspace`
- 默认采用邮箱密码模式，不启用第三方登录

### 5.2 用户基础资料自动补齐

注册成功后系统必须保证以下数据存在：

- `profiles`
- `quota_wallets`

实现方式优先采用数据库触发器或幂等服务端补齐逻辑，保证不会出现“auth 用户存在，但业务用户不存在”的断层。

### 5.3 受保护页面

以下路径需要真实会话：

- `/workspace`
- `/tasks`
- `/billing`
- `/recharge`
- `/account`
- `/admin`

但管理员路径还需附加数据库角色校验，不能仅凭存在会话就放行。

## 6. 积分、激活码与账本设计

### 6.1 钱包模型

钱包至少保留以下字段：

- `recharge_quota`
- `subscription_quota`
- `frozen_quota`

当前用户页面优先展示“当前可用积分”，即：

- `recharge_quota + subscription_quota`

冻结积分先用于内部扣费控制和后台显示。

### 6.2 激活码兑换

兑换接口改为：

- 前端只提交激活码
- 服务端根据当前会话解析当前用户
- 服务端调用数据库事务或 RPC 完成：
  - 校验激活码存在
  - 校验激活码未使用
  - 标记使用人和使用时间
  - 增加积分
  - 写入流水

### 6.3 扣费规则

- 创建任务前先检查余额
- 文章生成任务冻结 `500`
- 自动降AI任务冻结 `500`
- 任务成功后转正式扣费
- 任务失败时按规则解冻或返还

所有扣费、解冻、返还都必须在服务端完成，不能由前端决定。

## 7. 写作任务主流程设计

### 7.1 工作台统一体验

用户始终停留在工作台单页，页面内部展示不同阶段状态：

1. 上传文件与填写特殊要求
2. 文件解析
3. 主任务文件识别
4. 规则卡生成
5. 大纲生成与确认
6. 正文生成
7. 字数校正
8. 引用核验
9. 导出交付
10. 自动降AI

### 7.2 文件提取

支持格式：

- `txt`
- `md`
- `docx`
- `pdf`
- `ppt`
- `pptx`

系统必须提取正文，不允许只靠文件名判断。

提取后的文本用于：

- 任务书识别
- 规则卡生成
- 大纲生成
- 正文生成

### 7.3 主任务文件识别

识别优先级：

1. 显式规则信号匹配
2. AI 辅助判断
3. 无法稳定判断时，要求用户确认

系统不能在多个任务书候选时偷偷选一个。

### 7.4 规则卡

规则卡必须整合：

- 主题
- 目标字数
- 引用格式
- 章节数要求
- 必答问题
- 评分重点
- 用户特殊要求

优先级：

1. 主任务文件硬要求
2. 用户输入的特殊要求
3. 背景材料提示
4. 默认值

默认值：

- `2000` 字
- `APA 7`

### 7.5 大纲生成与确认

系统先生成：

- 英文主大纲
- 中文对照版

大纲要求：

- 独立文章标题
- 与字数匹配的章节数
- 每章主要内容说明
- 每章 3-5 个重点

默认章节数按“每 500 字一章”估算，除非任务书或用户明确指定。

大纲生成后进入等待确认状态：

- 用户确认后进入正文
- 用户修改则重新生成
- 建议最大修改次数：5 次

### 7.6 正文、字数、核验与交付

正文要求：

- 全英文
- 有独立标题
- 有章节结构
- 有 `References`
- 严格遵循已确认大纲

字数校正：

- 只校正文，不动 `References`
- 页面区分正式版本字数与候选版本字数

核验报告：

- 不伪装成全文阅读
- 基于标题、年份、DOI、链接、摘要等公开信息判断
- 输出清楚说明，不细分复杂等级

交付物：

- 最终版文章 Word
- 引用核验 PDF

### 7.7 自动降AI

用户点击“自动降AI”后：

- 只提取正文部分
- 调用 StealthGPT
- 输出“降AI后版本（Word）”
- 页面继续展示微信二维码与“人工降ai 请联系客服”

## 8. 后台与运营设计

### 8.1 后台模块

后台最少提供：

- 用户管理
- 激活码管理
- 任务管理
- 文件管理
- 财务/积分概览

### 8.2 后台权限

后台权限以 `profiles.role` 为准：

- `user`
- `admin`

所有后台页面与后台 API 都必须重新校验管理员角色。

### 8.3 后台人工操作能力

需支持以下人工操作：

- 手动补积分
- 手动扣积分
- 手动冻结/解冻用户
- 手动作废激活码
- 手动重试任务
- 手动延长文件保留期
- 手动删除已过期文件

### 8.4 审计日志

所有管理员关键操作写入审计日志：

- 操作人
- 目标用户
- 目标任务
- 操作内容
- 时间

## 9. 数据保留与清理策略

### 9.1 保留 3 天的数据

- 上传原始文件
- 中间大纲
- 草稿正文
- 交付文档
- 核验报告
- 降AI版本

### 9.2 长期保留的数据

- 用户账号资料
- 钱包余额
- 激活码记录
- 积分流水
- 任务摘要记录
- 后台操作日志

### 9.3 过期后的用户体验

- 任务记录仍显示
- 下载按钮不可用并提示“文件已过期”
- 后台仍可看到摘要和状态

## 10. 当前代码缺口总结

当前版本与目标状态之间的主要差距：

- 登录注册未接真实会话
- 余额显示未接真实钱包
- 激活码接口仍接受前端 `userId`
- 管理员权限仍基于普通 cookie
- 下载接口仍基于前端 `userId`
- 任务创建、大纲确认、主任务文件确认接口是 501 占位
- 文件正文提取只真实支持 txt/md
- 工作台主链路尚未落真实持久化
- 后台大量页面仍是 mock 数据

## 11. 实施顺序

推荐分六个阶段补完：

1. 真实登录、注册、钱包、激活码、安全权限
2. 工作台创建任务、上传文件、提取正文、识别主任务文件
3. 规则卡、大纲生成、修改与确认
4. 正文生成、字数校正、引用核验、Word/PDF 导出
5. 自动降AI、交付页、下载权限、3天过期清理
6. 后台真数据、人工操作、审计日志、最终上线检查

## 12. 风险与规避

必须规避以下风险：

- 一边改主项目，一边继续开发历史前端目录
- 前端继续传 `userId`
- 扣费与任务创建分离，导致账务与任务不一致
- 管理员权限继续依赖伪造 cookie
- 下载接口缺少归属校验
- 先删旧文件再开发，导致回滚困难
- 把所有长步骤塞进单次请求中，导致超时和难重试

## 13. 结论

本项目不需要推翻重做。现有主项目已经具备可复用的前端设计、数据表基础、导出能力和部分业务骨架。

接下来应当采用“保留前端，补齐真实后端”的策略，按优先级完成：

- 认证与钱包
- 安全与权限
- 任务主流程
- 交付与降AI
- 后台真数据与人工管理能力

这样可以在最小视觉变更和最小架构风险下，把拼代代PDD补成真正可售卖、可运营、可维护的正式产品。
