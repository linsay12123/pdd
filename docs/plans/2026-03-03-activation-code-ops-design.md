# 拼代代PDD 激活码中控与兑换闭环设计方案

## 1. 背景与目标

当前产品已经明确切换到“激活码充值”模式，不再走在线支付。  
为了让系统可持续运营，需要把这条链路做成完整闭环：

1. 管理员可以稳定地批量生成激活码并管理库存。
2. 用户可以在额度中心直接兑换，并拿到明确结果。
3. 同一个激活码只能成功使用一次，不能重复使用。
4. 所有关键动作都可追踪（谁生成、谁兑换、何时发生）。

## 2. 已确认业务规则（冻结）

以下规则来自本轮确认，后续实现必须严格遵守：

1. 开发优先顺序：先把后台做强，再回头补齐前台兑换体验。
2. 兑换失败提示：要明确显示原因（不是统一模糊报错）。
3. 管理员单次批量生成上限：50 个。
4. 后台激活码列表：完整显示激活码，不打码。
5. 激活码档位固定：1000 / 5000 / 10000 / 20000。
6. 积分消耗固定：
   - 生成文章：500
   - 自动降AI：500

## 3. 范围边界

### 3.1 本次范围（In Scope）

1. 激活码后台中控（生成、查询、筛选、导出、复制）。
2. 用户兑换接口与兑换结果回传（成功/失败原因）。
3. 兑换原子性保障（防双花：同码双请求只能成功一次）。
4. 额度中心页面接入真实兑换接口与积分刷新。
5. 管理员鉴权与关键操作审计。

### 3.2 非本次范围（Out of Scope）

1. 恢复在线支付（Stripe/支付宝/微信/加密货币）功能。
2. 新增套餐档位和浮动扣费策略。
3. 大规模权限系统重构（仅使用当前管理员标识机制）。

## 4. 架构方案

## 4.1 数据层设计

新增持久化表（Supabase/Postgres）：

1. `activation_codes`
   - `id` uuid pk
   - `code` text unique（完整激活码）
   - `tier` integer（仅允许 1000/5000/10000/20000）
   - `quota_amount` integer
   - `status` text（`unused` / `used`）
   - `created_by_user_id` uuid
   - `used_by_user_id` uuid nullable
   - `created_at` timestamptz
   - `used_at` timestamptz nullable

2. 兑换日志沿用现有：
   - `quota_ledger_entries`：记录积分变更（新增 `activation_credit`）
   - `admin_audit_logs`：记录后台生成行为

说明：
- 前期内存仓储已可跑流程，但不满足生产可靠性。  
- 本次闭环要以数据库为准，保证重启不丢数据。

## 4.2 核心流程

### A. 管理员生成激活码

1. 管理员提交 `tier + count`。
2. 后端校验：
   - 角色必须管理员
   - `count` 在 1~50
   - `tier` 在四档内
3. 生成随机码（例如 `PDD-5000-XXXXXXXX`）。
4. 批量落库为 `unused`。
5. 记录 `admin_audit_logs`。
6. 返回完整码列表给前端（支持复制/导出）。

### B. 用户兑换激活码

1. 登录用户提交激活码。
2. 后端校验格式。
3. 数据库内做“查码 + 状态变更 + 加积分 + 写账本”的原子操作。
4. 返回：
   - 成功：到账积分、最新余额
   - 失败：明确原因

## 4.3 API 设计

管理员接口：

1. `POST /api/admin/activation-codes/create`
   - 入参：`tier`, `count`
   - 出参：完整激活码数组

2. `GET /api/admin/activation-codes/list`
   - 查询：`status`, `keyword`, `page`, `pageSize`
   - 出参：分页列表

3. `GET /api/admin/activation-codes/export`
   - 查询同列表筛选
   - 输出 CSV 文件流

用户接口：

4. `POST /api/quota/redeem-code`
   - 入参：`code`
   - 成功出参：`creditedQuota`, `currentQuota`
   - 失败出参：明确错误信息

## 4.4 错误语义（用户可见）

兑换失败仅暴露以下可控文案：

1. 激活码不存在
2. 激活码已被使用
3. 激活码格式不正确
4. 系统繁忙，请稍后再试

## 4.5 鉴权与安全

1. 用户兑换接口：必须已登录。
2. 管理员接口：必须管理员身份。
3. 激活码只允许后端生成，不接受前端自造。
4. 关键操作写审计日志（谁、何时、做了什么）。

## 5. 前端设计

## 5.1 后台中控页（先完成）

分为四块：

1. 总览卡片：今日发码、今日兑换、未用库存、累计兑换积分。
2. 发码区：档位选择 + 数量（<=50）+ 生成按钮。
3. 列表区：完整显示激活码，支持筛选与搜索。
4. 操作区：复制全部、导出 CSV。

## 5.2 用户额度中心（后完成）

1. 输入激活码 + 兑换按钮。
2. 成功后更新余额与记录。
3. 失败显示明确原因。
4. 处理中禁用按钮，防重复提交。

## 6. 一致性与并发策略

为避免“双花”（同码并发兑换两次），兑换流程必须使用数据库原子更新：

1. 仅对 `status=unused` 的码执行更新到 `used`。
2. 更新成功后再写积分与账本。
3. 若状态已变更，直接返回“已被使用”。

## 7. 测试策略

1. 单元测试：
   - 码生成唯一性
   - 档位与数量校验
   - 失败原因映射
2. 接口测试：
   - 管理员接口鉴权
   - 兑换成功/失败分支
3. 并发测试：
   - 同一码双请求仅成功一个
4. 页面测试：
   - 后台生成与列表刷新
   - 用户兑换后余额刷新

## 8. 交付标准（Definition of Done）

满足以下条件才算完成：

1. 后台可批量生成（最大50）、完整展示、筛选、导出。
2. 用户端可兑换并明确展示失败原因。
3. 同一码不可二次兑换（并发下也成立）。
4. 所有关键测试通过（lint / unit / build）。
5. 文档更新到位（运行说明与运营手册）。
